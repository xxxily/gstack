# GStack Browser V0 — AI 原生开发浏览器

**日期：** 2026-03-30
**作者：** Garry Tan + Claude Code
**状态：** Phase 1a 已交付，Phase 1b 进行中
**分支：** garrytan/gstack-as-browser

## 核心论点

其他每个 AI 浏览器（Atlas、Dia、Comet、Chrome Auto Browse）都是从消费级浏览器出发，在上面叠加 AI。GStack Browser 反其道而行。它以 Claude Code 作为运行时，给它一个浏览器视口。

Agent 是首要的公民。浏览器是画布。Skill 是一等公民功能。你不是"在使用一个带 AI 帮助的浏览器"。你在使用一个能看到网页并与之交互的 AI。

这是后 IDE 时代的 IDE。代码在终端里。产品在浏览器里。AI 同时在两者之间工作。Cursor 为文本编辑器做的事，GStack Browser 为浏览器做。

## 今天它是什么（Phase 1a，已交付）

一个可双击运行的 macOS .app，封装了 Playwright 的 Chromium，内置 gstack sidebar 扩展。你打开它，Claude Code 就能看到你的屏幕、导航页面、填写表单、截图、检查 CSS、清理覆盖层，运行任何 gstack Skill。无需碰触终端。

```
GStack Browser.app（389MB，189MB DMG）
├── 编译后的浏览二进制文件（58MB）— CLI + HTTP 服务器
├── Chrome 扩展（172KB）— sidebar、活动流、检查器
├── Playwright 的 Chromium（330MB）— 实际的浏览器
└── 启动脚本 — 绑定项目目录，设置环境变量
```

启动 → Chromium 打开并带 sidebar → 扩展自动连接到浏览服务器 → Agent 就绪约 5 秒。

## 它将成为什么

### Phase 1b：开发者 UX（下一步）

**命令面板（Cmd+K）：** 标志性交互方式。打开一个模糊过滤的 Skill 选择器。输入 "/qa" 开始 QA 测试、"/investigate" 用于调试、"/ship" 创建 PR。Skill 从浏览服务器获取而非硬编码。面板是一切功能的入口。

**快速截图（Cmd+Shift+S）：** 捕获当前视口并将其推送到 sidebar 聊天中，附带 "你看到了什么？" 上下文。AI 分析截图并给出可操作的反馈。一个按键即可生成视觉 bug 报告。

**状态栏：** 每个页面底部固定的 30px 条。显示 Agent 状态（空闲/思考中）、工作区名称、当前分支和自动检测的开发服务器。点击开发服务器 pill 即可导航。始终可见的上下文显示 AI 在做什么。

**自动检测开发服务器：** 启动时扫描常用端口（3000、3001、4200、5173、5174、8000、8080）。如果恰好找到一个服务器，自动导航到它。状态栏中显示开发服务器 pill，一键切换。

### Phase 2：BoomLooper 集成

sidebar 连接到 BoomLooper 的 Phoenix/Elixir API，而非本地 `claude -p` 子进程。BoomLooper 提供：

- **多 Agent 编排。** 并行启动 5 个 Agent，每个拥有自己的浏览器标签页。一个运行 QA，一个做 design review，一个监控回归。
- **Docker 基础设施。** 每个 Agent 获得一个隔离的容器。容器内的浏览器测试开发服务器。无端口冲突，无状态泄漏。
- **会话持久化。** Agent 对话在浏览器重启后依然存在。从上次离开的地方继续。
- **团队可见性。** 你的队友可以实时观看你的 Agent 在做什么。就像结对编程，但结对的是 5 个 AI Agent，而你是指挥。

### Phase 3：Browse 作为 BoomLooper 工具

浏览二进制文件成为 BoomLooper 中的 MCP 工具。Docker 容器中的 Agent 使用浏览命令测试开发服务器、截图、填写表单、验证部署。需要跨平台编译（linux-arm64/x64）。

### Phase 4：Chromium Fork（触发条件门控）

当扩展侧面板遇到硬性 API 限制时，GStack Browser 交付给外部用户、构建基础设施存在、业务证明维护成本合理时：fork Chromium。采用 Brave 的 `chromium_src` 覆盖模式，CC 驱动的 6 周 rebases（CC 约 2-4 小时 vs 人类 1-2 周）。约修改 20-30 个文件。

### Phase 5：原生 Shell

SwiftUI/AppKit 应用外壳，带原生 sidebar、隔离的 Chromium 服务。完整的平台集成。如果 Phase 4 的 Chromium Fork 包含了原生 sidebar，可能会被该阶段取代。

## 愿景：AI 浏览器能做什么

### 1. 看你所看

浏览器是 AI 的眼睛。不仅通过截图（虽然它能做到），还通过 DOM 访问、CSS 检查、网络监控和可访问性树解析。AI 理解页面结构，而不仅仅是像素。

**现在：** `snapshot` 命令返回任何页面的可访问性树表示。AI 能"看到"每个按钮、链接、表单字段和文本元素。元素引用（`@e1`、`@e2`）让 AI 可以点击、填写和交互。

**下一步：** 实时页面观察。AI 注意到页面变化、控制台出现错误、网络请求失败。无需询问即可主动调试。

**未来：** 视觉理解。AI 比较前后截图来捕获视觉回归。像素级的设计审查。"这个按钮向左移动了 3px，字体从 14px 变成了 13px。"

### 2. 对所见的操作

不只是读取页面，而是像人类用户一样与之交互。

**现在：** 点击、填写、选择、悬停、输入、滚动、上传文件、处理对话框、导航、管理标签页。全部通过浏览服务器的简单命令。

**下一步：** 多步用户流。"登录，去设置，修改时区，验证确认消息。" AI 在每个步骤中链接命令并进行验证。

**未来：** 自主 QA Agent。"测试这个页面上的每个链接。填写每个表单。试着打破它。" AI 运行穷尽式交互测试，无需脚本。发现人类测试员会遗漏的 bug，因为它尝试了人类想不到的组合。

### 3. 浏览时同时编写代码

这是关键差异点。AI 能同时看到浏览器中的 bug 并在代码中修复它。

**现在：** sidebar 聊天连接到 Claude Code。你说"这个按钮错位了"，AI 读取 CSS，识别问题并提出修复方案。`/design-review` Skill 截图、识别视觉问题并提交带前后对比证据的修复。

**下一步：** 实时重载循环。AI 编辑 CSS/HTML，浏览器自动重载，AI 视觉验证修复。简单的视觉修复无需人类干预。"修复这个页面上的每个间距问题"变成一个 30 秒的任务。

**未来：** 全栈调试。AI 在浏览器中看到 500 错误，读取服务器日志，追踪到失败的代码行，编写修复并在浏览器中验证。一条命令："这个页面坏了。修好它。"

### 4. 理解整个技术栈

浏览器不只是视口。它是窥探应用健康的窗口。

**现在：**
- 控制台日志捕获 — 每个 `console.log`、`console.error` 和警告
- 网络请求监控 — 每个 XHR、fetch、WebSocket 和静态资源
- 性能指标 — Core Web Vitals、资源计时、绘制事件
- Cookie 和存储检查 — 读写 localStorage、sessionStorage
- CSS 检查 — 计算样式、盒模型、规则级联

**下一步：**
- 网络请求重放 — "用不同参数重放这个失败的请求"
- 性能回归检测 — "这个页面比昨天慢了 200ms"
- 依赖审计 — "这个页面加载了 47 个第三方脚本"
- 可访问性审计 — "这个表单没有标签，这些颜色对比度不达标"

**未来：**
- 完整应用遥测 — 实时监控 CPU、内存、GPU 使用
- 跨浏览器测试 — 同一测试套件在 Chrome、Firefox、Safari 上运行
- 与真实用户监测关联 — "这个 bug 影响了 12% 的线上用户"

### 5. 工作区模型

浏览器本身就是工作区。不是工作区中的一个标签页。而是工作区本身。

**现在：** 每个浏览器会话绑定到一个项目目录。sidebar 显示当前分支。状态栏显示检测到的开发服务器。

**下一步：** 多项目支持。在不关闭浏览器的情况下切换项目。每个项目有自己的标签页集合、自己的 Agent、自己的上下文。就像 VSCode 工作区，但是为浏览器设计的。

**未来：** 团队工作区。多个开发者共享一个浏览器工作区。看到彼此的 Agent 在工作。协作调试，一个人导航，另一个人观看 AI 实时修复。

### 6. Skill 作为浏览器能力

每个 gstack Skill 都成为浏览器能力。

| Skill | 浏览器能力 |
|-------|-------------------|
| `/qa` | 测试每个页面，发现 bug，修复它们，验证 |
| `/design-review` | 截图 → 分析 → 修复 CSS → 再截图 |
| `/investigate` | 在浏览器中看到错误 → 追踪到代码 → 修复 → 验证 |
| `/benchmark` | 测量页面性能 → 检测回归 → 告警 |
| `/canary` | 监控线上站点 → 定期截图 → 变更告警 |
| `/ship` | 运行测试 → 审查 diff → 创建 PR → 浏览器验证部署 |
| `/cso` | 在真实浏览器中审计页面的 XSS、开放重定向、点击劫持 |
| `/office-hours` | 浏览竞品网站 → 综合观察 → 设计文档 |

命令面板（Cmd+K）是中心。你不需要知道 Skill 的存在。输入你想要的，模糊过滤器找到合适的 Skill，然后 AI 以浏览器为上下文运行它。

### 7. 设计循环

AI 驱动的设计是一个循环，而不是一次交接。

```
生成 mockup（GPT Image API）
  → 在浏览器中审查（与线上站点对比）
  → 用反馈迭代（"把头部调高"）
  → 确定方向
  → 生成生产级 HTML/CSS
  → 在浏览器中预览
  → 用 /design-review 微调
  → 交付
```

浏览器弥合了"在 Figma 中的样子"和"在生产中的样子"之间的差距。因为 AI 能同时看到两者。

### 8. 安全循环

CSO 审查在真实浏览器中进行，而不只是静态分析。

- 向每个输入字段注入 XSS payload，检查是否执行
- 通过从不同来源重放请求来测试 CSRF
- 通过导航到精心构造的 URL 检查开放重定向
- 验证 CSP 头部是否真正被执行（不只是存在）
- 通过实时操纵 cookie 和 token 来测试认证流程
- 通过在 iframe 中加载站点来检查点击劫持

静态分析捕获模式。浏览器测试捕获现实。

### 9. 监控循环

部署后的 canary 监控，在真实浏览器中进行。

```
部署 → 浏览器加载线上 URL
  → 截图基线
  → 每 5 分钟：截图、对比、检查控制台
  → 告警条件：视觉回归、新的控制台错误、性能下降
  → 检测到严重错误时自动回滚
```

带 AI 判断的综合监控。不只是"页面是否返回 200"，而是"页面看起来是否正确且运行正常。"

## 竞争格局

| 浏览器 | 方法 | 差异化 | 劣势 |
|---------|----------|---------------|----------|
| **Atlas** | Chromium fork + AI 层 | 自主 Agent 浏览器，"OWL" 隔离 Chromium | 消费级导向，无代码集成 |
| **Dia** | AI 原生浏览器 | 干净的 UI，为 AI 交互设计 | 无开发工具，无代码编辑 |
| **Comet** | AI 浏览器 | 多 Agent 浏览 | 早期阶段，开发工作流不确定 |
| **Chrome Auto Browse** | 扩展 | Google 自有，深度 Chrome 集成 | 仅限扩展，无代码编辑 |
| **Cursor** | VSCode fork + AI | 最佳的代码编辑 | 无浏览器视口 |
| **GStack Browser** | CC 运行时 + 浏览器视口 | 在浏览器中发现 bug，在代码中修复，验证 | 目前仅 macOS，无消费级功能 |

GStack Browser 不与消费级浏览器竞争。它与在浏览器和编辑器之间切换的工作流竞争。目标是让这种切换变得不可见。
