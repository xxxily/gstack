# 系统化调试

## 铁律

**无根本原因调查之前，不进行修复。**

修复症状会导致打地鼠式调试。每个没有解决根本原因的修复都会让下一个 bug 更难找到。找到根本原因，然后修复它。

---

## Phase 1：根本原因调查

在形成任何假设之前收集上下文。

1. **收集症状：** 阅读错误消息、堆栈跟踪和复现步骤。如果用户没有提供足够的上下文，**一次只问一个问题**。不要一次问五个问题。

2. **阅读代码：** 从症状回溯到潜在原因追踪代码路径。搜索所有引用，阅读故障点周围的逻辑。

3. **检查近期变更：**
   ```bash
   git log --oneline -20 -- <affected-files>
   ```
   之前能正常工作吗？什么变了？回归意味着根本原因在 diff 中。

4. **复现：** 你能确定性地触发这个 bug 吗？如果不能，在继续之前收集更多证据。

5. **检查 memory** 中是否有针对同一区域的之前的调试 session。同一文件中反复出现的 bug 是一种架构气味。

输出：**"根本原因假设：……"** ——关于什么出了错以及为什么的具体、可验证的声明。

---

## Phase 2：模式分析

检查这个 bug 是否匹配已知模式：

**竞态条件**……间歇性、依赖于时间。检查对共享状态的并发访问。

**Nil/null 传播**……NoMethodError、TypeError。可选值上缺少守卫。

**状态损坏**……数据不一致、部分更新。检查事务、回调、hooks。

**集成失败**……超时、意外响应。外部 API 调用、服务边界。

**配置漂移**……本地正常、staging/prod 失败。环境变量、feature flags、DB 状态。

**过期缓存**……显示旧数据、清除缓存后修复。Redis、CDN、浏览器缓存。

同时检查：
- 项目中是否有相关问题的已知问题
- git log 中同一区域的之前的修复。同一文件中反复出现的 bug 是一种架构气味，不是巧合。

**外部搜索：** 如果 bug 不匹配已知模式，在线搜索该错误类型。**先脱敏：** 剥离主机名、IP、文件路径、SQL、客户数据。搜索错误类别，而不是原始消息。

---

## Phase 3：假设验证

在写**任何**修复之前，验证你的假设。

1. **确认假设：** 在疑似根本原因处添加临时日志语句、断言或调试输出。运行复现。证据是否匹配？

2. **如果假设是错的：** 搜索错误（先脱敏敏感数据）。回到 Phase 1。收集更多证据。不要猜。

3. **3-strike 规则：** 如果 3 个假设都失败了，**停止**。告诉用户：

   "已测试 3 个假设，都不匹配。这可能是架构问题而非简单的 bug。"

   选项：
   - 继续调查新的假设（描述它）
   - 升级到人工审查（需要熟悉系统的人）
   - 添加日志并等待（对该区域加以仪器化，下次捕获它）

**红旗**……如果你看到任何这些，放慢速度：
- "先快速修复一下"……没有"先"这回事。要么修复好，要么升级。
- 在追踪数据流之前就提出修复……你在猜。
- 每个修复都在其他地方暴露新问题……是层级错了，不是代码错了。

---

## Phase 4：实现

一旦根本原因被确认：

1. **修复根本原因，不是症状。** 消除实际问题的最小变更。

2. **最小 diff：** 最少触及的文件，最少的变更行。忍住冲动，不要重构相邻代码。

3. **写一个回归测试：**
   - **不加修复时失败**（证明测试是有意义的）
   - **加了修复后通过**（证明修复有效）

4. **运行完整的测试套件。** 不允许回归。

5. **如果修复涉及 >5 个文件：** 在继续之前向用户标记爆炸半径。这对外 bug 修复来说太大了。

---

## Phase 5：验证与报告

**全新验证：** 复现原始 bug 场景并确认它已修复。这不是可选的。

运行测试套件。

输出结构化的调试报告：

**调试报告**
- **症状：** 用户观察到的内容
- **根本原因：** 实际出了什么问题
- **修复：** 变更了什么，带文件引用
- **证据：** 测试输出、复现显示修复有效
- **回归测试：** 新测试的位置
- **相关：** 同一区域的之前 bug、架构注释
- **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED

将报告保存到 `memory/`，带今天的日期，以便未来的 session 可以引用。

---

## 重要规则

- **3+ 次失败的修复尝试：停止并质疑架构。** 是架构错了，不是假设失败了。
- **永远不要应用你无法验证的修复。** 如果你不能复现并确认，不要发布它。
- **永远不要说"这应该能修复它。"** 验证并证明它。运行测试。
- **如果修复涉及 >5 个文件：** 在继续之前向用户标记。
- **完成状态：**
  - DONE……找到根本原因，应用修复，写了回归测试，所有测试通过
  - DONE_WITH_CONCERNS……已修复但无法完全验证（例如，间歇性 bug，需要 staging）
  - BLOCKED……调查后根本原因不清楚，已升级
